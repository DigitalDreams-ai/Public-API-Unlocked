# Mapping Format Reference

All mapping values must use this format:

`$<ObjectApiName>.<FieldApiName>`

## Valid Examples

- `$PublicApi_Submission__c.Status__c`
- `$PublicApi_Submission__c.Received_At__c`
- `$PublicApi_Submission__c.CreatedDate`
- `$Lead.Email`
- `$Account.Name`

## Invalid Examples

- `$submission.Email__c`
- `$intake.Email__c`
- `Email__c`
- `$PublicApi_Submission__c`

## Configuration Shape

Payload mappings are stored separately on:

- `Inbound_Payload_Configuration__c`
- `Outbound_Payload_Configuration__c`

Each configuration stores an array of mapping rows:

```json
[
  { "externalKey": "status", "value": "$PublicApi_Submission__c.Status__c", "sortOrder": 1 }
]
```

Related-record rules are stored on:

- `Related_Record_Configuration__c`

Each rule row uses this shape:

```json
[
  {
    "targetLookupField": "AccountId",
    "relatedObjectApiName": "Account",
    "matchFieldName": "Name",
    "sourceKey": "accountName",
    "createIfMissing": true,
    "createMappings": [
      { "sourceKey": "phone", "fieldName": "Phone" }
    ],
    "onMultipleMatches": "Error"
  }
]
```

## Validation Rules

- Inbound submission processing requires a non-empty mapping configuration.
- Outbound status delivery requires a non-empty mapping configuration.
- Field references must match `Target_Object_API_Name__c`.
- Unknown object or field references are rejected by validation.
- Invalid JSON config is treated as a configuration error.
- Related-record rules validate:
  - lookup field exists on the target object
  - related object exists
  - match field exists and is queryable/filterable
  - create mappings use writable related-object fields
  - `onMultipleMatches` is one of `Error`, `UseOldest`, or `UseNewest`
- Outbound source-object selection only controls payload/query behavior. Automatic invocation still requires either:
  - the built-in `PublicApi_Submission__c` trigger, or
  - a record-triggered Flow that calls `PublicApiOutboundStatusAction`

## Failure Behavior

- Inbound config errors return HTTP `400`.
- Related-record config errors return HTTP `400` before target-record insert.
- Outbound config errors are logged as failed webhook deliveries and do not abort the entire queueable batch.
